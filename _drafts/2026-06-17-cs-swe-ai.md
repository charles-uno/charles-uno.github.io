First, a quick note on terminology. Computer science covers the logical, mathematical, and physical foundations of computing. Topics include hardware architecture, cryptography, and movement of traffic across a network. Software engineering, on the other hand, is about building a product. Writing code to implement the logic on a computer is a tool along the way. 

Ten years ago, that distinction didn't much matter. Companies of all sizes were scrambling to hire software engineers as the world became [increasingly computer-dependent][software_eats_world]. On top of that, they were trying to hire another layer of software engineers to implement the internal [infrastructure and processes][history_of_devops] popularized[^devops] by big tech. Not to mention the demand added during COVID, when the whole world was suddenly online. Job seekers with the ability to write code -- computer science graduates and [bootcampers][bootcamp] alike -- saw strong salaries and extravagant benefits.

The US had [fewer than a million][2010_swe_count] software engineers in 2010. By 2020, the number was [nearly 1.5 million][2020_swe_count]. The rapid increase was largely fueled by an increased supply of computer science graduates. The number of students majoring in computer science [increased by a factor of five][fivefold_increase] between 2008 and 2024.

Now the winds are shifting. Software engineering job postings peaked during the COVID pandemic; they've been [decreasing][swe_jobs_dropping_post_covid] for the past few years. Layoffs are [increasing][2026_tech_layoffs]. Salaries are [dropping][swe_salary_dropping]. Not only that, but AI has suddenly widened the gap between computer science and software engineering. Coding is increasingly delegated to AI agents while human engineers focus on system design and product behavior.

These changes raise a few big questions:
- Should computer science education follow the software engineering industry?
- If so, what skills should we prioritize moving forward?
- And what fundamentals should be depreioritized to make room?

I was ambivalent to AI tooling until early 2026; now it's good enough that I choose to use it for work on a daily basis. This is how things look from where I'm standing.


## Software Isn't All Equally Soft

Some software engineers work very close to the hardware. Anyone who works on a compiler, a device driver, or an operating system kernel needs to have an intimate knowledge of the underlying machine. This knowledge was important when they were writing their code by hand, and it will still be important as the line-by-line syntax is increasingly delegated to AI agents. 

Other software engineers are focused on the product. For example, when a software engineer is building a website for a local restaurant, they care about how the different colors and fonts and photos line up with one another. They don't particularly need to care about how those instructions are being implemented at the hardware level.

Most of us live somewhere in the middle. For example, last year, I worked on inbox filtering for large Instagram accounts. These filters made it easier for content creators and celebrities to wade through the thousands of messages they get every day. I didn't touch the app itself; that was the product engineers. My job was to build nice interfaces for the product engineers to query the information they needed: eligibility tracking, parsing updates from message metadata[^metadata], and privacy policies to ensure appropriate access. In turn, my work was built on lower-level abstractions related to scalability, observability, and database configuration.

[^metadata]: Notably: I do not have access to the content of your messages! Metadata means I can see who is sending the message, which allows me to mark the message read/unread as appropriate. I can see if it's a group thread. I can see the type of message (text, photo, reaction, etc). I can see if the participants are verified. Stuff like that. 

AI and vibe coding allow *way* more people to use code to solve problems without worrying about the machine. This sort of software engineering does not require a background in computer science. Software engineers working at a high level would be better served learning the fundamentals of product management so they can effectively translate domain expertise into product requirements.

Before AI, software engineers were mostly working on layer 5. Layer 4 was nice to know, useful once in a while. Below there you get diminishing returns. With AI, software engineers are mostly living on layer 6. Layer 5 is useful once in a while. 

CS *could* go back to being a niche major. There will always be a demand for Linux admins. Someone has to write the compilers and design the chips. I doubt this is what you want, but I do want to call it out as an option.

If we want CS to stay big, the program probably needs to maintain itself as an entrypoint to software engineering. That means leaning into skills that are important to AI-based software engineering, and leaning away from less-relevant fundamentals to make room.

These (in my opinion) are some areas to lean into:

- System design. Big-picture familiarity with the different components in a distributed system, as well as the constraints of thost components
- Verification. Static analysis, test-driven development, and validation criteria. Find problems fast to iterate fast
- Code defense. I don't care where you code came from. I want you to stand up and convince me that it's safe to deploy for a billion users
- DevOps concepts. AI can change thousands of lines of code in an instant. Agile/Scrum for scoping and coordination across a team. Pipelines, containers, and feature flags govern deployment. Telemetry, alerts, and oncall rotations are how you respond to problems that slip through
- Tech debt. Figuring out a legacy code base. Consolidating duplicate logic. Data and logic migrations
- Privacy and security. Responsible handling of user data. Vulnerabilities. Attack vectors
- Practice using AI tools! But in a way that doesn't short-circuit learning

These are some areas to lean away from:

- Specific syntax for different languages. It's ok to let AI and linters handle the brackets and semicolons
- Toy scripts. If AI can do it in a one-sentence prompt, there's not much value in asking a human to write it. Unless it helps solidify the concepts in their brain??
- ??? very hard to know what fundamentals are load-bearing ???

# How much has changed?

- Big picture, I'm still building software to move product goals forward. In my case, server-side logic for messaging on FB, IG, etc
- Looking at what I actually *do* day-to-day, my job is extremely different compared to six months ago
- If we're talking about a small change to a piece of code where I am already an expert, I can probably do it faster by hand
- Any time I have to write a big block of code, or touch boilerplate, or work with an unfamiliar API, or if there's any element of tedium, it's faster to use AI
- Comparable to moving from Assembly to C. The code quality *technically* is lower, but the time saved is so great that nobody cares
- Plus it's async. I can have two or three agents running at a time
- It is still sometimes wrong! Human reviews still exist, at least for now

# Why aren't we seeing this everywhere?

(Yet)

- Budget. Meta has given every engineer a blank check for AI tokens for months. I'm spending hundreds of dollars a week on this
- Infrastructure. Meta has cutting-edge automation up and down the stack. Strict rules for API endpoints. Static code analysis. Unit and integration tests. Level 1: this is a defense against accidentally committing junk. Level 2: AI can see failures and iterate on its code until they are fixed
- Scale. We have dedicated engineers working on wrappers and tooling and integrations for AI tools. I can talk to it like a peer and assume it already has baseline fluency on our code base, wiki, project management systems, etc

# What's different for entry-level engineers?

- Job interviews (at least at Meta) now use AI in coding interviews. It's about the tooling more than the algorithms
- Less demand for junior engineers. Expectation that they'll be replaced by seniors plus AI
- What happens when today's senior engineers want to retire? This is the billion-dollar question

# What skills are important?

- System design. If the AI code looks a bit sloppy but the tests all pass, so be it. If the AI introduces an unnecessary network call to a different component, that's an expensive mistake
- Creativity, design, judgment
- Context switching. Can you jump back and forth between AI sessions? This is hard!
