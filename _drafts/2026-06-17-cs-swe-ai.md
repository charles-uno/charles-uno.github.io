First, a quick note on terminology. Computer science covers the logical, mathematical, and physical foundations of computing. Topics include hardware architecture, cryptography, and movement of traffic across a network. Software engineering, on the other hand, is about building a product. Computers are a tool along the way. 

Ten years ago, that distinction didn't much matter. Companies of all sizes were scrambling to hire software engineers as the world became increasingly computer-dependent. On top of that, they were trying to hire another layer of software engineers to implement the internal processes popularized by big tech: agile methodology, release automation, live monitoring, etc. Job seekers with the ability to write code -- computer science graduates and bootcampers alike -- saw strong salaries and extravagant benefits.

The US had fewer than a million software engineers in 2010. By 2020, the number was nearly 1.5 million. The rapid increase was largely fueled by an increased supply of computer science graduates. The number of students majoring in computer science increased fivefold between 2008 and 2024. 

Now the winds are shifting. 
- Software engineer job posting hit an all-time high during COVID; they've been dropping since.
- Layoffs are increasing.
- AI has widened the gap between the fields of computer science and software engineering

Twenty years ago, CS was a niche major for people who liked computers. My graduating class (2009) had about a dozen CS majors. Last year, we had closer to sixty. The major presumably became large because it served as an entrypoint to lucrative jobs in software engineering. 

Recent advances in AI tooling have significantly changed what software engineering looks like.

Before AI, software engineers were mostly working on layer 5. Layer 4 was nice to know, useful once in a while. Below there you get diminishing returns. With AI, software engineers are mostly living on layer 6. Layer 5 is useful once in a while. 

CS *could* go back to being a niche major. There will always be a demand for Linux admins. Someone has to write the compilers and design the chips. I doubt this is what you want, but I do want to call it out as an option.

If we want CS to stay big, the program probably needs to maintain itself as an entrypoint to software engineering. That means leaning into skills that are important to AI-based software engineering, and leaning away from less-relevant fundamentals to make room.

These (in my opinion) are some areas to lean into:

- System design. Big-picture familiarity with the different components in a distributed system, as well as the constraints of thost components
- Verification. Static analysis, test-driven development, and validation criteria. Find problems fast to iterate fast
- Code defense. I don't care where you code came from. I want you to stand up and convince me that it's safe to deploy for a billion users
- DevOps concepts. AI can change thousands of lines of code in an instant. Agile/Scrum for scoping and coordination across a team. Pipelines, containers, and feature flags govern deployment. Telemetry, alerts, and oncall rotations are how you respond to problems that slip through
- Tech debt. Figuring out a legacy code base. Consolidating duplicate logic. Data and logic migrations
- Privacy and security. Responsible handling of user data. Vulnerabilities. Attack vectors
- Practice using AI tools! But in a way that doesn't short-circuit learning

These are some areas to lean away from:

- Specific syntax for different languages. It's ok to let AI and linters handle the brackets and semicolons
- Toy scripts. If AI can do it in a one-sentence prompt, there's not much value in asking a human to write it. Unless it helps solidify the concepts in their brain??
- ??? very hard to know what fundamentals are load-bearing ???

Some looser notes...

This is not going to happen all at once! TW and I are both coming from the perspective of big tech. Most software engineers are not working on the bleeding edge. Software engineers in the midwest are mostly working at places like Target and Best Buy. A ways back in terms of processes

Human code review. AI writes code fast. The quality is pretty good and getting better. Reviewing changes by hand is already a bottleneck, and becoming more so all the time

A quick aside: a few months back, there were concerns that software engineers would be replaced by AI. This has (mostly) not happened. I think we're in a Jevons paradox situation. AI makes cognitive work cheaper, but the demand for cognitive work is elastic enough to absorb the supply. Lots more people in lots more fields are going to start using computers (via AI) to solve problems

"Computer" used to be the name of a job. Now it's the name of a machine used by accountants. When AI does the software engineering, what will they call the people who write the prompts?

CS likes to focus on concepts rather than tools. Don't want to spend time learning (eg) Jenkins or GitHub Actions or specific test or metrics software. But AI also broadens the idea of what is a tool. Programming languages are tools. New ones can pop up. And with AI, new ones can potentially pop up *fast*

Historically, college is a place where rich people go to amuse themselves and launder their priveledge into credentials. That is less true now as it becomes more accessible. College is now much more closely tied to the job market

# How much has changed?

- Big picture, I'm still building software to move product goals forward. In my case, server-side logic for messaging on FB, IG, etc
- Looking at what I actually *do* day-to-day, my job is extremely different compared to six months ago
- If we're talking about a small change to a piece of code where I am already an expert, I can probably do it faster by hand
- Any time I have to write a big block of code, or touch boilerplate, or work with an unfamiliar API, or if there's any element of tedium, it's faster to use AI
- Comparable to moving from Assembly to C. The code quality *technically* is lower, but the time saved is so great that nobody cares
- Plus it's async. I can have two or three agents running at a time
- It is still sometimes wrong! Human reviews still exist, at least for now

# Why aren't we seeing this everywhere?

(Yet)

- Budget. Meta has given every engineer a blank check for AI tokens for months. I'm spending hundreds of dollars a week on this
- Infrastructure. Meta has cutting-edge automation up and down the stack. Strict rules for API endpoints. Static code analysis. Unit and integration tests. Level 1: this is a defense against accidentally committing junk. Level 2: AI can see failures and iterate on its code until they are fixed
- Scale. We have dedicated engineers working on wrappers and tooling and integrations for AI tools. I can talk to it like a peer and assume it already has baseline fluency on our code base, wiki, project management systems, etc

# What's different for entry-level engineers?

- Job interviews (at least at Meta) now use AI in coding interviews. It's about the tooling more than the algorithms
- Less demand for junior engineers. Expectation that they'll be replaced by seniors plus AI
- What happens when today's senior engineers want to retire? This is the billion-dollar question

# What skills are important?

- System design. If the AI code looks a bit sloppy but the tests all pass, so be it. If the AI introduces an unnecessary network call to a different component, that's an expensive mistake
- Creativity, design, judgment
- Context switching. Can you jump back and forth between AI sessions? This is hard!
